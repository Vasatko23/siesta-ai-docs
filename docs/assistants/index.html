Assistants

The Assistants section displays an overview of all created AI assistants within the platform. Each assistant is an independent unit with its own configuration, which can be further edited or developed. This page serves as a central place for their management, quick overview, and access to their details.

List of Assistants​

The main overview displays a list of all existing assistants. The table contains the following columns:

  • – unique assistant identifier

  • Name – assistant name (e.g., ESG Assistant, HR Assistant)
  • Instructions – a shortened preview of the instructions that define the assistant's behavior
  • Temperature – parameter determining the creativity of responses
  • Evolution – development status of the assistant based on feedback (e.g., No feedback, Possible)
  • Status – current status of the assistant (e.g., Active, Deactivate)
  • Actions – option to view details, edit, or perform other actions via the context menu

There is a Create Assistant button on the screen, which allows you to create a new assistant. After clicking it, the user can choose whether to create an assistant from scratch or use one of the prepared templates.

Creating an Assistant​

After clicking the Create Assistant button, a dialog window appears where the user can choose how to create the assistant. The user can choose to start with a blank form or use one of the available templates.

This option makes it easier to repeatedly deploy common types of AI agents and allows for a quick start without having to set up the assistant from scratch.

Key interface elements:

  • Search field – allows filtering templates by name or keywords
  • "Blank" option – create an assistant without preset values
  • Predefined templates – choose from prepared assistant configurations, e.g.:
    • Add Github contact
    • Structure Output Parser
    • Api Agent

Each template has a name and a brief description for quick orientation. By clicking on a specific option, the user is taken to the editing interface for detailed assistant configuration.

New Blank Assistant Form​

After choosing to create a new assistant, the user is presented with a form to define all key properties of the assistant. This form can also be used for later editing of an existing assistant.

The form is divided into several logical sections:

Details​

This section defines the basic characteristics of the assistant – its name, behavior description, and technical parameters. The user specifies:

  • Name – assistant name
  • Instructions – basic instructions defining its behavior, tone, and function
  • Model – language model selection (e.g., GPT-4o)
  • Credentials – choice of credentials (e.g., for access to external systems)
  • Initial Message – default message when starting a conversation
  • System Message – hidden instruction defining the assistant's system behavior

Data​

Here you set which data the assistant should work with. The platform allows you to connect various data sources, whether internal documents, company knowledge bases, or publicly available information. This way, each assistant can have its own specific context and access only to relevant data. Supported integrations include company knowledge systems, internal documents uploaded to the platform, API connections, or even publicly available sources published online.

Selection of data sources for the assistant:

  • Access to all data sources – access to all data within the organization
  • Limited data sources access – manual selection of specific connections

Model configuration​

Settings for parameters that affect the style and quality of outputs:

  • Response format – response format (e.g., Text)
  • Temperature – creativity of responses
  • Maximum Length – maximum output length
  • Presence Penalty – penalty for repeating already mentioned topics
  • Frequency Penalty – penalty for frequent word occurrences

This configuration allows you to finely tune the model's behavior according to requirements for tone, format, or output consistency.

Scheduler​

The user can schedule when and how often the assistant should be activated – for example, every Monday at 8:00 AM once a month:

  • Time – start time
  • Day – day of the week
  • Repetition – frequency (e.g., daily, weekly, monthly)

Scheduled activation allows assistants to be used even without direct user interaction.

Tools​

You can add your own tools to the assistant, such as integration with a calendar, internal systems, or API connectors. The assistant can then, for example, schedule a meeting, create a task, or trigger an action in another tool.

Integrations​

Through this section, the assistant can be connected to communication tools such as Slack or Microsoft Teams. Users can then communicate with the assistant directly from the environment they use daily, without having to launch the platform's web interface.

API​

Here you can make the assistant available via the API interface, including the option to copy the entire call structure, request body, and response. This functionality enables integration with your own systems or services that use the assistant as backend intelligence.

Public chat​

This section allows you to make the assistant publicly available, for example by embedding a simple JavaScript code into a website. The public chat does not require login and is therefore suitable for customer support, career pages, or client portals.

Saving an Assistant as a Template​

The platform allows you to save the currently created assistant as a template, which can then be reused when creating other assistants. This feature supports repeatability, consistency, and scalability of configurations across the organization.

After clicking the Save as Template button, a modal window appears where the user fills in:

  • Name – template name
  • Description – brief description of its focus and use

The saved template then appears in the overview when creating a new assistant and can be edited or reused at any time.

Slack Integration (Connect Slack Integration)​

Within the Integrations section, you can connect the assistant to an external communication platform such as Slack. This connection allows communication with the assistant directly in the environment users commonly use.

When setting up the integration, a modal window appears where you can fill in:

  • Webhook URL – address for sending messages to the Slack channel
  • Channel – target channel name
  • Name – assistant name in Slack
  • Icon URL – address for a custom icon

This option ensures smooth integration between AI agents and the organization's everyday tools without the need to switch to another interface.

Assistant Detail​

Each assistant has its own detailed interface, divided into several thematic subsections. These allow users to conveniently manage configuration, analyze performance, respond to feedback, or track change history.

Overview​

The Overview tab is a central panel that summarizes key information about the assistant on a single screen. Here, the user finds basic metrics on how often the assistant is used, what data it has connected, how much feedback it has received, and what its current development status is.

This overview allows you to quickly see how the assistant works, what feedback it receives, and who uses it most often, without having to go into individual detail tabs. It is the ideal starting point for an immediate assessment of the health and usage of each assistant.

Configuration​

The Configuration tab is for detailed assistant settings – its technical configuration, default behavior, and connection to other platform features. This section allows you to precisely define how the assistant should behave, what data it works with, which model it uses, and what tools it has available.

Here, the user sets the basic information about the assistant, including name, instruction description, system messages, and the model to be used (e.g., GPT-4o). Data access is also defined here – either global connection to all sources or limited access to selected datasets.

The configuration also includes the model layer, where you can adjust parameters such as response creativity (temperature), repetition penalty, or maximum output length. These options allow you to finely tune the assistant's behavior for a specific purpose or expectation.

In this section, you can also connect the assistant to tools (Tools) that it can actively use – for example, to connect to a calendar, API, or other systems. The entire configuration is designed so that the assistant can be quickly launched, edited, or versioned at any time.

Interfaces​

The Interfaces section allows you to set how the assistant will be made available to end users. Here you define how and where it will be possible to communicate with the assistant, whether internally, via API, or publicly through integration channels and widgets.

The user can activate the assistant for access via direct API calls, deploy it as a public chat on the web using a prepared script, make it available as an anonymous public chat via its own URL, or connect it to tools such as Slack, Microsoft Teams, WhatsApp, or email.

Each of these connections has its own specific settings and identifier. The platform also allows you to enable or disable availability for each interface as needed. This makes it possible to precisely control where the assistant is active and from where it receives requests.

Prompts​

The Prompts section is used to control the assistant's logic and behavior through so-called system instructions. Here, the user defines how the assistant should respond, what attitude to take, communication style, or response structure.

Each prompt represents a specific instruction block that the model receives before processing the user's input. The assistant uses it to determine what role to play, which information to prioritize, and what responses to generate.

Analytics​

The Analytics section provides a detailed view of the assistant's technical and operational parameters. Here, users will find summary statistics and visualizations that help understand how the assistant is used, what its load is, and how it performs over time.

Monitored metrics include the number of requests, volume of tokens consumed, ratio of input and output tokens, response processing speed, and other indicators. This information is available in the form of clear graphs and bar visualizations, allowing you to spot trends, fluctuations, or possible anomalies.

Analytics is an essential tool for administrators and product teams who want not only to operate but also optimize assistants. It helps evaluate when the highest usage occurs, what the consumption-to-performance ratio is, and how quickly the assistant responds in real conditions.

Evolution​

The Evolution section allows you to manage the development of the assistant's system instructions (prompts) based on real behavior and feedback. Administrators can compare different prompt versions, evaluate their impact, and apply changes in a targeted and controlled way.

The core part is a comparison of the currently active prompt with the proposed modification. The user sees the previous and new instruction side by side and can easily identify how the assistant's logic is changing. In addition to text comparison, the system also collects specific use cases to test the change.

Evolution also includes an overview of feedback that led to the change or to which the new prompt version should respond. This feedback includes ratings, comments, and the identity of the users who submitted them.

Thus, Evolution is a tool for improving assistant behavior based on data, not intuition. It enables continuous development, controlled testing, and documentation of all changes over time.

Conversations​

The Conversations section provides an overview of all interactions between users and a specific assistant. Here, users will find a list of conversations with information about the date, initiating user, subject, and total number of messages in the exchange.

This section allows administrators to look into the actual content of communication, analyze how the assistant is used in practice, and verify how queries were answered. The displayed information can be used for further assistant development, prompt tuning, or ensuring compliance with internal rules.

Each conversation includes the option for a detailed view. Users can retrospectively track the entire communication, including all steps and responses, increasing transparency and allowing for retrospective control if needed.

Feedbacks​

The Feedbacks section serves as a centralized overview of feedback provided by users on the assistant's responses. Each record includes the date, user identity, interaction result, and type of rating – for example, positive, negative, with a comment, or other specific evaluation.

Thanks to this section, administrators can easily identify responses that were inaccurate, misleading, or particularly helpful. All reactions are traceable, and you can look back at the context in which the rating was given.

This feature is key for future assistant development – it provides a data-driven basis for prompt adjustments, refining the data context, or evaluating the need for training. Combined with the Evolution section, it forms part of the continuous improvement of assistant outputs.

History​

The History section is for transparently tracking all important changes made to the assistant. Each record includes the date, author of the change, affected entity, and type of action performed.

This audit trail is important for operational oversight, security standards, and tracing interventions in environments where assistants often undergo development or modifications.
